# Security Audit Report - 2026-06-03

This is a security audit report created by Google Gemini on 2026-06-03
for the Best Practices Badge project.

## Executive Summary

A manual security audit of the BadgeApp codebase was conducted on June 3, 2026. The project demonstrates an exceptional level of security awareness and implementation. The "defense-in-breadth" approach described in the assurance case is consistently applied throughout the source code.

This report identifies one notable vulnerability (SSRF via project URLs) which has been remediated through a new "Strict Domain Only" policy. In practice this really isn't a "vulnerability" in the usual sense, because in our deployment there aren't any specially-visible sites we have special access to, but it's better to prevent problems in the first place.

## Vulnerabilities Identified

### VULN-001: Server-Side Request Forgery (SSRF) and Malformed URLs

- **Vulnerability Type**: Security
- **Severity**: Medium (prior to remediation), Mitigated (now)
- **Source Location**: `app/lib/evidence.rb:33`, `app/validators/url_validator.rb`
- **Description**: The application previously allowed user-provided `homepage_url` and `repo_url` to include IP addresses and hostnames without dots (e.g., `localhost`). While the data was used limitedly (checking security headers), it technically permitted the server to probe internal networks.
- **Investigation of Existing Data**:
    A database audit revealed that very few projects use IP-based URLs, and those that do illustrate the risk:
    1.  **ID 3733 (e-library)**: Homepage is `http://3.6.47.234`. This is an inactive personal project whose homepage is now down.
    2.  **ID 7829 (flosssearch)**: Homepage is `http://191.252.92.63/flosssearch`. This is an archived academic project whose IP has been hijacked and now hosts a Portuguese cryptocurrency dashboard.
    3.  **ID 8135 (watchtower)**: Uses a malformed URL `https://containrrr/watchtower` (missing dots in host).

    These findings confirm that **active, maintained OSS projects do not use IP-address domains** for their primary identities. IP-based URLs are highly unstable and prone to hijacking.
- **Remediation: Strict Domain Only Policy**:
    - **Rejection of all IP Addresses**: The application now forbids any IP address (IPv4/IPv6, hex, or integer formats) in project URLs. This eliminates an entire class of SSRF attacks and enforces professional standards.
    - **Mandatory Dot Check**: All hostnames must now contain at least one dot (`.`). This rejects local/internal hostnames and catches common malformed entries like `containrrr`.
    - **Dual Enforcement**:
        - **`UrlValidator`**: Rejects these URLs at the time of entry.
        - **`Evidence#get`**: Treats any existing dubious URLs in the database as leading to "no data" and logs a warning. This ensures the application never makes requests to dangerous hosts, even if they were previously allowed.
    - **Data Cleanup**: A database migration (`RemoveDubiousHomepageUrls`) was executed to automatically clear `homepage_url` values that are dubious, provided that a `repo_url` is also present. Before clearing the URL, the migration appends the original value to the project's description (e.g., "Old homepage_url: ...") to ensure data is preserved for administrative reference.
- **Impact**: Mitigated. The application now enforces robust domain-based validation, significantly hardening the system against SSRF, and existing data has been brought into compliance.

## Hardening Opportunities

### 1. Content Security Policy (CSP) - chartkick

In `app/controllers/project_stats_controller.rb`, the CSP is overridden to allow `'unsafe-inline'` for charting.
**Recommendation**: Explore using nonces with `chartkick` or moving chart initialization to separate JavaScript files.

### 2. Dependency Management

While dependencies are regularly updated, some gems are pinned for specific reasons.
**Recommendation**: Periodically review pinned dependencies to ensure no critical security patches are missed.

## Notable Security Controls (Strengths)

- **Translation Validation**: `test/models/translations_test.rb` validates ALL translation strings against a whitelist of HTML tags and attributes.
- **Timing Protection**: `User.authenticate_local_user` uses dummy hashes to prevent timing attacks.
- **Data Protection**: Sensitive information like email addresses and remember-me tokens are encrypted at rest.

## Conclusion

The BadgeApp demonstrates a mature security posture. The identified SSRF vulnerability has been addressed with a robust "Domain Only" policy that aligns with the practices of legitimate, maintained open source software projects.
